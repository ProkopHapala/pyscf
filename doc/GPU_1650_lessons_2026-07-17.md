**Question of the day:** can GPU XC on a GTX 1650 (4 GB, 48 KiB local, 14 CUs) compete with CPU `smallDFT` on PTCDA-scale molecules — and if not, what structural levers still move the needle?

---

## Executive summary

| Finding | Consequence |
|---------|-------------|
| Large **private** tile dumps in `rho_*_tiled` spill to DRAM | Fixing them **~3×** benzene OTF; PTCDA OTF almost unchanged (extra Hermite recomputes offset the win) |
| Tile / split-K tuning on 1650 is only **~1.5×** | Not enough to beat CPU cache on PTCDA |
| Dense all-atom work wastes most of PTCDA | **Screened radial** path: **~2.6×** vs OTF on PTCDA (`na≈21/38`, pair×≈5.8) |
| Best GPU XC on PTCDA/1650 today | **`production_radial_screened`** ≈ **1.07 s** gpuCL |
| Best overall production on this laptop | Still **CPU `smallDFT` + `ao_mode=cache`** ≈ **0.9 s** XC / cycle |
| Host buffer hygiene | Screen CSR must be **preallocated** (worst-case cap) + `enqueue_copy`, never sized/`COPY_HOST_PTR` from live lists |

## Hardware context (why 3090 recipes fail here)

| | GTX 1650 laptop | RTX 3090 (other doc) |
|--|----------------:|---------------------:|
| CUs | 14 | many more |
| Local mem / CU | **48 KiB** hard wall | larger effective headroom |
| Device DRAM | ~4 GB | 24 GB |
| Host RAM | 16 GiB, **0 swap** | 32 GiB class |
| Molecule that hurts | PTCDA `nao=286`, χ ≈ 3.5 GB | benzene was the tuning target |

Implications:

1. **`NATILE=4` / large `NPTILE×NATILE`** — ptxas rejects (shared > 48 KiB). Tile space is small.
2. Full GGA χ on GPU is unrealistic; OTF / radial are mandatory for large mols.
3. Host cannot hold two full χ tensors → Amdahl benches must run CPU and GPU in **separate processes**.
4. Split-K (3090 benzene winner) is **not** the 1650 winner — too little occupancy gain, extra reduce cost.

## Lesson 1 — Private memory is the silent killer

### Diagnosis

`rho_gga_tiled` held **per-thread private** dumps of entire i-atom tiles:

```text
*_tile[MAX_ITILE][...]   →  ~3840 B private / thread
```

On Turing (1650), that does **not** stay in registers. Compiler spills to **local/global**, destroying bandwidth and latency hiding. `__local` AO tiles in vmat (~16.5 KiB padded with `MAX_AO_ATOM=16`) are a separate issue — scarce but intentional.

User constraint (correct): **large private arrays sized by tile/atom/grid counts are unacceptable.** Local is precious (≤~8–16 KiB useful / WG on this GPU); private must stay tiny (scalars + tiny shells).

### Fix (existing tiled kernels, not a new path)

Stream **one i-shell / one i-atom at a time**: tiny private `Ri[MAX_SHELL]…`, recompute i-Hermite when needed. GGA rho: pack partials into `__local float4` instead of four arrays.

| Metric | Before | After |
|--------|-------:|------:|
| ρ private footprint | ~3840 B | **~96 B** |
| Benzene OTF gpuCL | ~241 ms | **~75 ms** |
| PTCDA OTF gpuCL | ~2.7 s | **~2.8 s** (flat / slightly worse) |

### Why PTCDA did not win from the spill fix alone

Dense OTF still walks **all atom pairs × all grid tiles**. After streaming, each pair pays **extra Hermite recomputes** (`× n_jTiles ≈ 19` on PTCDA). Spill removal helps benzene (small `n_jTiles`); on PTCDA the recompute tax cancels the DRAM win.

**Tried and rejected:** `VMAT_NPSUB=16` (more barriers) → PTCDA **regression**. Keep `VMAT_NPSUB = NPTILE`.

**Generalization:** fixing spills is necessary hygiene and can be huge on small systems; on large dense systems you still need **less work** (screening), not only cheaper work.

## Lesson 2 — Tile / profile tuning is a small lever on 1650

Benzene OTF tile quick-grid span: **~48–72 ms (~1.5×)**. Split-K band similar and **slower** than best OTF.

Hard fails: `NATILE=4`, `NPTILE=128`+`NATILE=2` (shared mem > 48 KiB).

**Do not** expect 3090’s `production_otf_radial_vmat_splitk` (~12 ms class on 3090 benzene) to be default here. On this laptop:

| Molecule size | Prefer |
|---------------|--------|
| Small (benzene) | OTF tiled (`64,2,256`) or hybrid OTF ρ + rad vmat |
| Medium–large (PTCDA) | **`production_radial_screened`** |
| Production SCF if RAM fits | **CPU smallDFT cache** |

## Lesson 3 — Screening is the multi× lever for PTCDA

### Physics / sparsity

Each atom has a radial cutoff `Rcut` from Hermite tails (`grid_screen.compute_atom_rcut`). A grid tile keeps only atoms whose sphere overlaps the tile AABB. Pairs further pruned by `Rcut_i + Rcut_j`.

PTCDA estimates (measured):

| Quantity | Dense | Screened |
|----------|------:|---------:|
| Active atoms / gTile | 38 | **~21** mean |
| Pair factor vs dense | 1× | **~5.8×** fewer |
| Benzene na_mean | 12 | ~11.5 (almost dense) |

So screening **must** help PTCDA and **cannot** help benzene much — exactly what the benches showed.

### New kernels (additive — do not replace OTF)

| Kernel | Role |
|--------|------|
| `rho_gga_radial_screened` | gTile WGs; active-atom CSR; `ia≤ja` with ×2 off-diagonal DM |
| `vmat_gga_radial_screened_pair` | `(ia,ja)` WGs; CSR `pair→gTile` lists |

Host:

- `grid_screen.build_gtile_atom_lists(..., pair_screen=True)` + invert to pair→gTile CSR
- Profile `production_radial_screened` (`rho_mode`/`vmat_mode` = `'radial_screened'`)
- Persistent radial `R,dR` on grid (setup once)

### Memory rule (host GPU buffers)

**Wrong:** `cl.Buffer(..., COPY_HOST_PTR, live_list.nbytes, live_list)` after screening (size depends on geometry every time).

**Right:** preallocate **worst-case** capacity once:

| Buffer | Capacity |
|--------|----------|
| `gtile_atom_off` | `n_gtile+1` |
| `gtile_atom_list` | `n_gtile * natoms` |
| `pair_gtile_off` | `n_pairs+1` |
| `pair_gtile_list` | `n_pairs * n_gtile` |

Fill with `enqueue_copy`. Caps live in `otf['screen_cap']`. Geometry moves → rebuild lists + copy, **no realloc**.

## Strategies that worked (ordered by impact on 1650 / PTCDA)

1. **Reduce asymptotic work** — atom/pair screening + upper-triangle DM in ρ (~2.6×).
2. **Kill private spills** — stream shells; keep private O(shell) (~3× benzene; hygiene everywhere).
3. **Precompute geometry-invariant radials** — `R,dR` once; gather in kernels.
4. **Device-resident XC** — GPU PBE + reduce (already ≪ ρ/vmat).
5. **Tile launch tuning** — only ~1.5×; do after structure is right.
6. **Overlap DF J with XC** — hides ~50–70 ms J; irrelevant until XC ≪ 200 ms.

Strategies that **did not** win on this GPU for PTCDA:

- Radial precomp / hybrid / split-K **without** screening (slower than plain OTF).
- More vmat NPSUB / barriers (regress).
- Hoping tile sweeps alone close the CPU gap.

## Best kernels / profiles **right now** (GTX 1650)

### Quick chooser

```text
                    ┌─ benzene / small mol ──► production_otf
                    │                         (or otf_radial_vmat if hybrid wins)
Need GPU XC? ───────┤
                    └─ PTCDA / sparse large ─► production_radial_screened

Need fastest SCF on this laptop + RAM for χ?
  └─ CPU smallDFT ao_mode='cache' + DF storage='incore'
```

### Kernel map

| Profile | ρ kernel | vmat kernel | When |
|---------|----------|-------------|------|
| `production_otf` | `rho_gga_tiled` / pair | `vmat_gga_tiled` / pair | default small/medium; no screen |
| `production_otf_radial_vmat` | OTF ρ | `vmat_gga_radial_precomp_*` | benzene hybrid (3090-era); mixed on 1650 |
| `production_otf_radial_vmat_splitk` | OTF ρ | split-K radial vmat | **3090** production; not 1650 default |
| **`production_radial_screened`** | **`rho_gga_radial_screened`** | **`vmat_gga_radial_screened_pair`** | **best GPU on PTCDA/1650** |

### Compile / launch defaults that matter

- `NPTILE=64`, `NATILE=2`, `WGS_VMAT=256` for OTF on 1650 benzene
- Stay under 48 KiB `__local`
- Private: no `MAX_ITILE`-sized dumps
- Screen buffers: prealloc + copy

## Outlook — future improvements (priority)

### P1 — Beat CPU cache on PTCDA GPU (~1.07 s → under 0.8 s XC)

| Idea | Why it should help | Risk |
|------|--------------------|------|
| **Tighter `screen_eps` / angular margin** | Fewer active atoms/pairs; measure `|vxc|` vs eps | Accuracy cliff |
| **Compact AO tiles** (`MAX_AO_ATOM` → real max shells) | Cut padded `__local` (~22% useful today) | Occupancy / bank conflicts |
| **Better pair→gTile locality** | Sort tiles; coalesce radial gathers | Host prep cost |
| **Symmetric vmat / upper-triangle only** | Mirror ρ’s `ia≤ja` | Race / reduce design |
| **Fused ρ+PBE+wv on device** | Already tiny; only after ρ/vmat | Low ROI now |

Target: another **~1.5–2×** on vmat (still ~73% of screened XC) without new spills.

### P2 — Setup / geometry updates

- Rebuild screen CSR on atom move: **copy into prealloc only** (done for alloc policy).
- Optionally refresh `R,dR` in-place the same way (already sized by `nradial×ngrids`).
- Cut OpenCL first-compile pain (~7 s profile apply) via binary cache (if not already reliable).

### P3 — CPU path still matters on this machine

Until GPU XC is under ~700 ms:

- Keep **`ao_mode=cache`** as production default when RSS fits (~4 GB with DF).
- `ao_mode=stream` for RAM-starved runs (slower multi-cycle).
- CPU ρ/vmat are still ~90% of best-path cycle — bandwidth / layout work there pays on the **winning** path today.

### P4 — Do not chase yet

- eig / DIIS / libxc (noise)
- DF J (~70 ms) until XC is much cheaper
- Blind port of 3090 split-K defaults
- Materializing full χ on the 1650

### P5 — Validation discipline

- Always report `|vxc|` vs NumInt (or vs OTF) with the same DM.
- Warm + CL event times; don’t mix 1650 and 3090 tables.
- Separate processes for CPU χ vs GPU when RAM is tight.
- Full stdout on long jobs (no silent pipes).

## Mental model (updated)

```text
XC time ≈  (#active atom-pairs) × (work per pair per gTile) × (mem penalty)

1650 day:
  mem penalty ↓↓  (kill private spills)     → big on benzene, weak on PTCDA alone
  #pairs ↓↓↓      (grid_screen + CSR)       → big on PTCDA, noop on benzene
  work/pair ↓     (radial precomp, tiles)   → secondary; can hurt if denser traffic
```

**Same code, different bottleneck by molecule size.** Profile choice must be **size-aware**, not cargo-culted from 3090 benzene.

## One-line takeaway

On GTX 1650, **stop spilling private tiles**, then **stop doing dense atom work** — screened radial is the best GPU XC path for PTCDA today, but **cached CPU smallDFT still wins production SCF** until another ~20–30% comes out of screened vmat (or screening gets tighter without losing parity).
